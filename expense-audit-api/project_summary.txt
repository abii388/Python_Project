Petty Cash Management Project Summary

1. Project Overview
- This is a Django backend for a petty cash management system.
- It exposes REST API endpoints for authentication, funds, transactions, audit tracking, notifications, and reporting.
- Authentication uses JSON Web Tokens (JWT) with `rest_framework_simplejwt`.
- The project supports PostgreSQL via `DATABASE_URL` and uses in-memory Django Channels for optional real-time messaging.

2. Installed Libraries
- Django==4.2.11: main web framework.
- djangorestframework: REST API framework.
- djangorestframework-simplejwt: JWT authentication.
- django-cors-headers: CORS support for frontend access.
- django-filter: filtering support for API querysets.
- psycopg2-binary: PostgreSQL database adapter.
- python-decouple: environment variable management.
- openpyxl: Excel file generation.
- channels: Django Channels for WebSocket support (in-memory layer configured).
- drf-spectacular: OpenAPI schema and Swagger documentation.

3. Apps and Responsibilities
- `apps.accounts`: user authentication, registration, JWT login/logout, password reset, and role management.
- `apps.funds`: fund creation, balance tracking, and permission-based fund management.
- `apps.transactions`: petty cash transaction creation, listing, and role-based access to transaction data.
- `apps.audit`: audit logging and retrieval for sensitive actions and approval events.
- `apps.notifications`: notification delivery and optional real-time messaging using Channels.
- `apps.reports`: export generation, Excel reports, and analytics endpoints.

4. Security and Authentication
- Default REST framework permission is `IsAuthenticated`.
- JWT authentication is enabled via `rest_framework_simplejwt.authentication.JWTAuthentication`.
- Tokens are issued by `/api/accounts/login/` and refreshed by `/api/accounts/refresh/`.
- Public endpoints (no authentication required):
  - `POST /api/accounts/register/`
  - `POST /api/accounts/login/`
  - `POST /api/accounts/refresh/`
  - `POST /api/accounts/reset-password/`
- All other API endpoints require a valid `Authorization: Bearer <access_token>` header.

5. User Roles and Access Rules
- Application roles include `Admin`, `Cashier`, `Approver`, and `Auditor`.
- `Admin` can manage funds, view all transactions, audit logs, and generate reports.
- `Auditor` can view audit logs, transactions, and fund activity, but cannot modify core fund settings.
- `Approver` can approve individual transactions (there is an `approve` endpoint and `CanApproveTransaction` permission).
- Full replenishment requests and multi-step approval workflows are planned but not implemented in this version.
- `Cashier` creates transactions and sees only their own records.

6. URL Access Summary
- `/api/accounts/register/` - public registration.
- `/api/accounts/login/` - public login.
- `/api/accounts/refresh/` - public token refresh.
- `/api/accounts/logout/` - authenticated users.
- `/api/accounts/reset-password/` - public password reset request.
- `/api/accounts/profile/` - authenticated users.

- `/api/funds/` - authenticated users can list funds; create/update/delete restricted to Admin.
- `/api/transactions/` - authenticated users; Admin/Auditor see all transactions; Approver and Cashier see scoped data.
- `/api/audit/` - authenticated Admin and Auditor access only.
- `/api/reports/` - authenticated users with access to export and filtering endpoints.
- `/api/notifications/` - authenticated users receive notification data and real-time events.

7. Overall Application View
- Users sign in with JWT credentials and use access/refresh tokens for API calls.
- The backend routes requests through app-specific APIs organized by responsibility.
- Admins manage funds while Cashiers create transactions; Admins and Approvers can approve individual transactions; replenishment and multi-step approval workflows remain planned but are not currently active.
- Audit logs capture important actions for compliance and review.
- Reports and exports are generated by `apps.reports`; notifications are managed by `apps.notifications`.
- The system is configured in `backend/config/settings.py` and environment variables are loaded from `.env`.

8. Configuration Files
- `backend/config/settings.py` contains database, JWT, Channels, and REST framework settings.
- `.env` stores environment values such as `SECRET_KEY`, `DEBUG`, and `DATABASE_URL`. Redis-related settings are present but currently inactive in the codebase.
- `backend/requirements.txt` lists the Python dependencies.

9. Notes
- Django Channels is configured with an in-memory channel layer by default.
- Redis and Celery are not currently active in the deployed codebase.
- Approval and replenishment workflows are planned features but are not implemented in this version.
- OpenAPI documentation is available via DRF Spectacular if schema endpoints are enabled.
- The admin interface is available at `/admin/` for superuser access.
